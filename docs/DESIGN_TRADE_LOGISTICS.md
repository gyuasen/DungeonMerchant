# 都市間運搬システム設計書 + 世界観と既存システムの対応

> ユーザー（開発者本人）が策定した設計文書（2026-07-16記録）。
> 世界観の正本は `docs/WORLDVIEW.md`。実装時は本書の「実装優先順位」に従うこと。

## 確定済みのスコープ決定（2026-07-17 ユーザー決定）

- **素材鑑定所は実装しない**: アイテムに状態などの個別ステータスが必要になり、データ圧迫につながるため。世界観文書に登場するが、ゲーム機能としては対象外。
- **旧「商会本部」は「商会組合」へ改名**: 設定上、この施設は主人公の商会ではなく、結界都市の商会組合（worldviewの「商人組合」と同一施設）。主人公は組合の窓口を利用する。
- **商会組合は商人の仕事を行う場所**: 傭兵管理（既存）に加え、**輸送部隊の編成・護衛設定・出発管理**を行う（下記「輸送部隊システム v1」）。
- **町マップの「編成所」は削除**: 傭兵の編成と確認はグローバルメニューから行う（メニューには既に編成・傭兵一覧が存在する）。
- 需要カテゴリの実装は案A（`WorldMapService`の静的テーブル、itemType粗粒度+アイテム個別上書き）で開始。需要倍率は0.8〜1.3程度に制限し、交易がダンジョン収益を無効化しないようにする。
- 「蘇生」系の表記は「**再活性治療**」へ統一する。

## 輸送部隊システム v1 設計（2026-07-17確定、優先度2〜3を統合した形）

主人公のパーティーとは**別の輸送部隊**を商会組合で編成し、護衛を付けて他町へ商品を送るシステム。輸送は自動解決（手動戦闘なし）で、プレイヤーが探索している間も並行して進行する。

### コアモデル（`TransportManager` + セーブv22）

- **輸送部隊（Convoy）**: 出発町／目的町（解放済み・進行順経路）、積荷（通常アイテムのみ、装備インスタンスは対象外）、護衛（最大3人、探索パーティー外の雇用傭兵、`InstanceId`参照）、残り日数
- **経路と日数**: 進行順位置の差=セグメント数、1セグメント=1日
- **出発時**: 在庫から積荷を差し引き、輸送費（セグメント数×基本費、兵站能力で軽減）を支払う
- **日送り（`DayManager.DayChanged`）で進行**: 毎日襲撃判定。護衛戦力（能力×レベルの簡易スコア）vs 経路危険度（通過地域基準）。撃退時は損失なし+護衛に少量経験値、突破された場合は積荷の一部（10〜50%）を損失し護衛のHPが減少（HP1で下限、死亡・ロストなし）
- **到着時**: 積荷を**到着町の需要倍率込み売値で自動売却**し、Gへ変換（=運送利益）。結果は日次リザルトへ表示
- **護衛の拘束**: 輸送中の傭兵は探索パーティーへ編成不可。一覧に「輸送任務中」を表示
- **セーブ**: `GameSaveData` Version 22。積荷はpersistentId+数量、護衛はinstanceId
- 乱数は注入可能な`Func<float>`（`BattleRewardService`と同パターン）でテスト決定性を確保

### UI（商会組合ページ）

- 既存の傭兵一覧に加え、「輸送部隊」セクションを追加: 部隊の新規編成（目的地選択→積荷選択→護衛選択→費用確認→出発）と進行中部隊の状況表示（目的地・残り日数・積荷・護衛）

### v1で意図的にやらないこと

- 手動戦闘（襲撃はすべて自動判定）／定期便（自動再出発）／御者・荷馬車契約による積載量拡張／装備インスタンスの輸送／輸送日数の兵站短縮 — いずれもv2以降の候補

## 町別倉庫システム設計（2026-07-17 ユーザー決定・確定）

輸送部隊の実装を受け、倉庫を全体共有から**町ごと**へ進化させる。あわせて商会組合からの遠隔操作と、傭兵の所在概念を導入する。

### 確定した設計決定（ユーザー回答）

1. **装備（個体）も町別倉庫にする**（アイテムと一貫。装備も輸送・交易の対象になる。傭兵への装備変更は「現在町の倉庫にある装備」のみ可能）
2. **遠隔売却は手数料なし、ただし販売タイムラグあり**: 商会組合から他町倉庫の売却指示を出すと、即時ではなく後日約定する（組合が現地で売り捌く）。タイムラグは町間距離（セグメント数、最低1日）。約定価格は**約定日のその町の需要込み売値**（日次変動リスクあり）。現在町の倉庫は従来どおり即時売却
3. **傭兵の町間移動はパーティー同行+輸送護衛のみ**（単独移動指示はv2候補）

### コア設計

- **MerchantInventoryを町別バケット化+現在町ファサード**: 内部を町indexごとの在庫（アイテム+装備個体）に分割し、既存の公開API（GetItemAmount/AddItem/TryAddItem/EquipmentInstances等）は「現在町の倉庫」を操作するファサードとして維持 → 市場・鍛冶・強化・戦利品・装備着脱の既存呼び出し箇所は無変更で「現在町の倉庫」セマンティクスに切り替わる
- 横断アクセス用の新API（townIndex指定版）を追加（組合の全町倉庫ビュー・遠隔売却・輸送搬入用）
- **倉庫容量は町ごとに独立**（現在の容量段階が各町に適用される）
- **鍛冶・強化は現在町の倉庫の素材のみ使用**（ファサードで自動的に成立。「どの町で加工するか」が意思決定になる）
- **輸送部隊の到着処理を「自動売却」から「目的地倉庫への搬入」へ変更**（搬入は容量超過を許容=組合の仮置き。ただし満杯時は新規取得のブロックは従来どおり）。売るかどうかは搬入後にプレイヤーが決める（現地で即時 or 遠隔売却指示）
- **傭兵の所在町**: 編成外傭兵は町に滞在。雇用=現在町。パーティー編成は現在町にいる傭兵のみ。プレイヤーの町移動完了でパーティー全員の所在を更新。輸送護衛は出発町にいる傭兵のみ選択可、到着で目的地に滞在（=護衛の回収・配置が経営判断になる）。治療は現在町の傭兵のみ。契約費・自然回復は所在に関わらず発生
- **図鑑・入手履歴は全体共有のまま**（記録であり物品ではない）
- **セーブ**: v23=町別倉庫+輸送搬入（既存在庫は移行時に現在町へ全量配置）、v24=遠隔売却注文+傭兵所在（既存傭兵は現在町へ）

### UI方針

- 在庫ページ=現在町の倉庫（タイトルに町名を表示）
- 商会組合の輸送オーバーレイに「全町倉庫」ビューを追加: 町ごとの在庫一覧+遠隔売却指示（約定予定日と予想額を表示）
- 売却約定・輸送搬入は日次リザルトに表示

## 現在のゲームサイクル

1. 結界都市で傭兵を雇う
2. 装備・治療・編成を整える
3. 街道やダンジョンへ派遣する
4. 魔物素材・装備・ゴールドを獲得する
5. 市場で売却するか、鍛冶素材として使用する
6. 傭兵や商人を成長させる
7. 日付を進め、借金を返済する
8. より危険な地域・都市・ダンジョンへ進出する

すでに「商人が傭兵を管理して利益を生み出す」という中心部分は十分実装されている。
今後追加する都市間運搬は、この循環に**物流**を加えるシステムになる。

## 世界観と既存システムの対応

### 商人と傭兵の契約（既存3種の設定づけ）

#### 日雇い
探索や街道護衛を一日だけ依頼する契約。傭兵側は自由度が高い一方で、商会から受けられる保障は少ない。
- 毎日の契約料が必要
- 治療費は一部のみ商会負担
- 装備は原則本人所有
- 危険な依頼を断られやすい

#### 臨時契約
一定期間、主人公の商会に所属する契約。ゲームで最も一般的な傭兵契約。
- 契約期間中は安定した仕事を得られる
- 装備や治療を商会が支援する
- 探索と輸送の両方へ派遣できる
- 契約終了後は自由になる

#### 専属契約
主人公の商会を主な雇用主とする契約。成功率判定は「傭兵側が商会の信用や待遇を見て判断している」と説明できる。
- 治療費や装備費を広く商会が負担
- 高危険度地域へ派遣できる
- 商会の専属護衛として主人公を守る
- 他商会の仕事を原則受けない
- 商会の信用が低い段階では契約してくれない

これにより、戦えない主人公が魔大陸で生きられる理由は「専属・臨時契約した傭兵が、商会の戦力であると同時に主人公自身の護衛でもある」からと設定できる。

### HPと治療システムの設定づけ

- HPは肉体の生命力ではなく、戦闘職に発現する**対外防護値**。軽傷・中傷・重傷の治療段階は肉体の傷ではなく**防護機構の損耗度**。
- HP0 = 防護停止・意識喪失・戦闘不能。死亡ではない。パーティーが全滅しない限り町へ生還できる。
- **UI表記の変更方針: 現在の「蘇生」は「再活性治療」へ改名する**（「蘇生」は一度死亡したように見えるため世界観とずれる）。
- 全滅時のペナルティ候補: 戦利品の大部分喪失／高額な救助費（救難組合設定）／全員の再活性治療費／探索日数の追加消費。装備耐久は現在存在しないため無理に追加しない。

### 地域構造（既存実装の設定づけ）

- **東方平原地域**: 低〜中級。生活用品が安い供給地。最初の事業地域。
- **北西山岳森林地域**: 上級。輸送が難しく、鉱石・魔石など武具素材の産地。この段階から運搬システムの重要性が上がる。
- **南西黒土地帯**: 最高級。特殊個体が多く、1億G達成の主な収益源。
- **中央島アステラ**: 魔力集中の特殊地域。完全なエンドコンテンツ（現行の3解放条件のまま）。

## 都市間運搬システム設計

### 基本設計

既存の街道移動（隣接町のみ・3〜5連戦・続行/撤退・街道別背景・レア遭遇・地域別敵編成）を基礎とし、「輸送依頼」を追加する。新しい大規模戦闘システムは作らない。

**商会輸送**（通常移動との差分）:
- 積荷が存在する
- 護衛傭兵を編成する
- 撤退時に積荷を一部失う
- 到着時に運送利益が発生する
- 重い積荷ほど敵との遭遇率が上がる
- 兵站能力が費用や損失率へ影響する

### 段階的解放

| 段階 | 内容 | 時期 |
|---|---|---|
| 第1段階: 直接輸送 | 傭兵パーティーが商品を直接運ぶ。毎回編成・街道戦闘・到着後販売。輸送中は探索に出せない | 序盤 |
| 第2段階: 運送契約 | 街道を一定回数突破すると御者・荷運び人と契約可。積載量増加・基本輸送費・護衛のみ自前・兵站で費用軽減 | 中盤 |
| 第3段階: 定期運搬経路 | 信用条件を満たすと定期便を設定可。一定日数ごと自動出発・結果は日次リザルト表示・襲撃時のみ手動戦闘可・維持費 | 終盤 |
| 第4段階: 商会物流網 | 複数都市を接続。往路に生活用品、復路に魔物素材の往復交易 | 最終盤 |

### 既存4能力との連動（新能力値は作らない）

| 商人能力 | 運搬への効果 |
|---|---|
| 交渉 | 都市別販売価格、運送業者との契約費 |
| 統率 | 護衛傭兵の契約維持、輸送中の士気 |
| 鑑定 | 運ぶべき素材の価値判断、特殊商品発見 |
| 兵站 | 積載量、輸送費、移動日数、襲撃時の損失軽減 |

兵站は既に倉庫容量・探索費用に影響しているため、輸送積載量・定期便維持費・商品損失率・輸送日数・消耗品消費量を追加すると存在感が強くなる。

### 市場システムとの連動: 都市別需要傾向

日ごとの価格変動（短期相場）と別に、都市特性（長期需要）を設定する。

| 都市特性 | 高く買う商品 |
|---|---|
| 港湾都市 | 高級素材、人類大陸向け輸出品 |
| 森林都市 | 毒腺、薬草系素材、魔物臓器 |
| 交易都市 | 加工品、装備、一般素材 |
| 山岳都市 | 食料、治療薬、木材 |
| 黒土地帯の都市 | 高級装備、消耗品、生活用品 |

→「森林都市では常に毒腺の需要が高いが、今日は在庫過多で価格が少し下がっている」という市場を実現する。

## 実装優先順位

| 優先度 | 内容 |
|---|---|
| 1 | **都市ごとの需要設定**: 現在の市場価格に都市特性による補正を追加。まず「別都市へ運ぶ意味」を作る |
| 2 | **商品を持った街道移動**: 既存の街道移動に積荷情報（出発都市/到着都市/積荷/数量/仕入れ価格/輸送費/到着時予想価格）を追加 |
| 3 | **輸送結果**: 到着・撤退・敗北に応じた積荷処理 |
| 4 | **運搬経路の開通**: 街道攻略回数や特殊依頼による経路登録 |
| 5 | **定期便**: 直接輸送が完成してから自動化を追加（市場・日付・セーブ・傭兵拘束・自動戦闘結果を同時に扱うため影響範囲が大きい。最後に回す） |

**総括**: 現在のDungeonMerchantは機能不足というより、実装済みの各機能を世界観と経営目的で束ねる段階に入っている。都市間運搬は新しい独立要素ではなく、既存の市場・街道・傭兵・兵站・日付・依頼システムを接続する機能として設計する。
